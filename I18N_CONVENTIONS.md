# WCPOS.com Internationalization Conventions

## Package choice

Use the existing `next-intl` stack for the Next.js website. Do not add another React translation framework unless there is a specific requirement `next-intl` cannot meet.

## Rule

All user-facing strings must be translatable. Hard-coded English is only allowed when it is documented as existing debt in `scripts/i18n/hardcoded-english-allowlist.json` or is an approved exception.

Approved exception categories:

- brand/product/proper names: `WCPOS`, `WooCommerce`, `WordPress`, `GitHub`, `PayPal`, `Stripe`, etc.;
- stable technical/legal tokens: `REST API`, `GPL`, `ABN`, version numbers, currency codes;
- dynamic user-generated or third-party content;
- internal logs and operator diagnostics not shown to users.

## Translation boundaries

Use `getTranslations` in server components/pages and `useTranslations` in client components. Keep namespaces aligned with product areas:

- `common`
- `header`
- `footer`
- `auth.*`
- `checkout.*`
- `account.*`
- `downloads.*`
- `roadmap.*`
- `support.*`
- `legal.*`
- `errors.*`

## Hidden and generated surfaces are in scope

The site is not fully translated unless these non-obvious surfaces are covered too:

- PDF receipts generated by `src/lib/pdf-receipt.ts`;
- email/alert HTML generated in `src/lib/sinks/email-sink.ts`;
- page metadata and Open Graph descriptions;
- `aria-label`, `alt`, `title`, and `placeholder` attributes;
- form validation and checkout recovery errors;
- API error messages that are displayed in the browser;
- empty states, loading states, toast messages, and status badges;
- account admin/impersonation banners;
- consent/privacy UI;
- release/download labels and roadmap status labels.

## Dates, numbers, currency, and receipts

Dates must be localized and unambiguous for international audiences. Avoid numeric-only display such as `02/01/2021`.

Use a shared helper for human date display. The default date shape should include a named month, for example:

- `17 Feb 2026`
- `Feb 17, 2026`
- localized equivalents for the active locale

PDF receipts must use the same locale-aware formatting rules as account pages. Receipt labels such as `Order date`, `Generated`, `Subtotal`, `Total`, `No tax has been added`, and footer/support text must come from translation messages before the PDF receipt surface can be considered complete.

Currency must use `Intl.NumberFormat` with locale and currency. Do not concatenate currency symbols manually for user-facing amounts.

## Rich text and links

Do not split translated sentences around links if word order can vary. Prefer rich-text messages with placeholders/functions, or sentence-level messages that include link slots.

Bad:

```tsx
<>Read our <Link href="/privacy">privacy policy</Link>.</>
```

Preferred:

```tsx
t.rich('privacyNotice', {
  link: (chunks) => <Link href="/privacy">{chunks}</Link>,
})
```

## Plurals and interpolation

Use ICU/plural-aware messages where counts are involved. Keep interpolation variable names stable across all locales.

Examples:

- activation counts;
- item quantities;
- release counts;
- machine counts;
- support result counts.

## Scanner guardrail

Run:

```bash
pnpm i18n:hardcoded
```

This compares current hard-coded English candidates against the baseline allowlist. New or increased candidates fail the check.

When extracting strings in a phase PR:

1. move strings into `messages/en.json` and matching locale files;
2. update components/pages to use `next-intl`;
3. run `pnpm i18n:hardcoded:update` to shrink the allowlist;
4. review the generated diff and ensure no new English debt was added.

The allowlist is debt, not permission to add more English.
